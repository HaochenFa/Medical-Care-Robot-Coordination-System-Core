\begin{abstract}
Medical Care Robot Coordination System (MCRoCS, ``McRocks'') is a lightweight operating-system core that coordinates medical-care robots with safe concurrency. It implements exactly the three Project Blaze services: a thread-safe task queue, exclusive zone access control, and a heartbeat-based health monitor. The main challenge was preserving correctness under concurrent threads while keeping the synchronization logic small enough to explain and test. The design uses Rust's \texttt{Mutex} and \texttt{Condvar} for shared-state coordination, while atomics are reserved for simulation metrics and validation \citep{rustmutex,rustcondvar}. The CLI demo deterministically shows concurrent task requests, non-overlapping zone occupancy, and an offline robot detected after missed heartbeats. Fresh release-mode benchmarks report throughput, zone-wait latency, and CPU time across contention, scalability, and offline scenarios; tests verify no duplicate task consumption, zone exclusivity, and timeout detection. The result is a minimal, explainable coordination layer aligned with the project's operating-systems objectives.
\end{abstract}

\section{Introduction}
Modern hospitals deploy autonomous robots for delivery, sanitation, and patient assistance. In these environments, coordination failures are more than inconveniences --- they cause safety incidents, resource conflicts, and service delays. MCRoCS addresses this by building a minimal OS core that enforces safe concurrency around three services: a task queue, a zone access controller, and a health monitor. The design deliberately excludes preemption and complex scheduling to keep the scope bounded to synchronization correctness, which is the stated learning outcome.

The project maps directly onto three OS fundamentals. First, multiple robot threads share the task queue, the zone ownership table, and the heartbeat registry; each structure requires correct mutual exclusion so that no two threads read inconsistent state. Second, synchronization via \texttt{Condvar} lets threads block without spinning, keeping CPU idle while they wait for a task or a free zone. Third, coordination among threads produces a deterministic, observable outcome: each task executes exactly once, each zone has at most one occupant, and a robot that stops sending heartbeats is detected and marked offline.

Removing preemption and advanced scheduling was a deliberate trade-off. It bounds the problem to what matters at this stage: proving that shared state is always consistent and that the concurrency invariants hold under load. A more complex scheduler would obscure those proofs rather than strengthen them. The minimal design also enables exhaustive unit testing --- each invariant has a direct test that exercises the boundary condition without relying on timing luck. Beyond testing, the bounded scope means every system behavior is observable in a short run: the CLI demo completes in under two seconds and shows concurrent task fetching, zone exclusivity, and an offline robot in sequence, mapping the three required OS concepts to three visible events.

The implementation is a Rust Cargo project organized around three core modules and a simulation driver. Each module owns its synchronization primitives and exposes a narrow interface, so the only shared state is the state each module explicitly manages. Rust's ownership model prevents data races at compile time \citep{klabnik2022}, reducing the remaining correctness questions to logical races, which the test suite targets directly. A separate integration test checks that the binary's standard output contains the expected fields for each mode, making correct behavior verifiable at the command-line level. The CLI supports a deterministic demo and configurable benchmark and stress modes, so the behaviors described in this report can be reproduced from the command line.

\section{Related Works}
Concurrency control has been a central operating-systems concern since early multiprogramming. Seminal work by Dijkstra and Lamport established that explicit coordination is required whenever multiple execution contexts share resources \citep{dijkstra1965,lamport1978}. Birrell's practical guide on thread programming translated those theoretical foundations into engineering guidance that influenced decades of concurrent software \citep{birrell1989}. Monitors and condition variables, introduced in Mesa, formalized a practical pattern: guard shared state with a mutex, and let threads wait on conditions that are re-evaluated after waking \citep{hoare1974,lampson1980}. MCRoCS uses this pattern directly --- \texttt{Mutex}-guarded state paired with \texttt{Condvar} waits --- in the task queue and zone controller alike \citep{rustmutex,rustcondvar}. Standard OS texts adopt the same emphasis on correct locking, short critical sections, and explicit wait conditions \citep{arpaci2018,silberschatz2018,tanenbaum2015}.

Task queues appear throughout systems software: thread pools, work-stealing schedulers, and I/O dispatch loops all rely on them \citep{arpaci2018,silberschatz2018}. High-performance systems often use lock-free or multi-queue designs to reduce contention, but those designs are significantly harder to reason about \citep{herlihy2020}. For this project, correctness and clarity take precedence over throughput. A single FIFO queue behind a mutex and condvar gives strong guarantees: tasks are consumed at most once, and idle consumers block rather than spin \citep{hoare1974}. This is the producer-consumer model from classical synchronization literature, and it is a well-understood pattern for teaching the subject \citep{downey2016}. Practitioner texts likewise recommend explicit synchronization on shared state in concurrent programs \citep{williams2019}.

Resource coordination often reduces to mutual exclusion over a namespace of shared resources \citep{silberschatz2018,tanenbaum2015}. The zone controller is conceptually a lock table: each zone has at most one owner at a time, and ownership must be unambiguous \citep{gray1993}. This resembles both OS resource maps and database lock managers, where a fixed ownership table makes current state explicit and auditable. In this implementation, zones are stored in a direct-indexed slot table sized at construction time, which keeps ownership queries to an array lookup with no hash overhead. Waiting on a per-zone condvar aligns with monitor-based coordination patterns \citep{hoare1974,lampson1980}. The coarse-grained lock simplifies correctness reasoning while the narrow critical section --- just a slot check and update --- limits how long the lock is held, reducing contention \citep{herlihy2020}.

Health monitoring through heartbeats is standard in distributed systems and fault-tolerant designs \citep{lynch1996,chandra1996}. A watcher accumulates periodic liveness signals from each node and marks nodes offline once a timeout expires. The accuracy of failure detection depends on timeout selection: shorter windows detect failures faster but increase false positives, while longer windows reduce false alarms at the cost of detection latency \citep{lynch1996}. MCRoCS adopts a simple timeout-based scheme suitable for a single-process simulation. The monitor stores a last-seen timestamp per robot and marks any robot overdue by a configurable threshold, illustrating the same trade-off highlighted in classic failure detector models \citep{chandra1996}. The connection to OS-level watchdog timers is direct: a kernel uses similar periodic checks to detect hung processes, facing the same fundamental tension between detection speed and false-positive rates.

Compared with lock-free queues, adaptive timeouts, or priority-aware scheduling, the project intentionally selects minimal mechanisms. This aligns with both the pedagogical goal and a common strategy in OS education: classic texts present mutex-plus-condvar as the foundational concurrency tool before introducing read-write locks or lock-free structures \citep{arpaci2018,tanenbaum2015}. Starting from the simplest correct mechanism produces a system whose behavior can be fully explained at the source-code level. Straightforward, well-documented primitives make it easier to state the safety invariants, trace each invariant to its enforcement code, and write tests that verify the boundary conditions without relying on timing artifacts --- which is more useful for demonstrating OS concepts than a faster but harder-to-reason-about alternative.

\section{Implementation}
\subsection{Architecture overview}
MCRoCS is organized around a CLI entry point and three execution modes --- demo, benchmark, and stress. The three-module split reflects a deliberate design choice: each OS concern (task dispatch, zone ownership, liveness tracking) has its own synchronization state, its own invariants, and its own test surface. Combining them under a single lock would simplify some call sites but would couple unrelated state, widen critical sections, and make it harder to isolate correctness failures in tests. Keeping the modules independent means that a bug in zone acquisition cannot corrupt heartbeat state, and vice versa.

The simulation driver (\texttt{sim.rs}) wires the three components, spawns robot threads, and collects metrics. Shared ownership uses \texttt{Arc} so each thread holds a reference to the same service instances without copying their state \citep{rustarc}. The \texttt{log\_dev!} macro emits structured logs in debug builds and compiles to a no-op in release, keeping benchmark output clean. Figure~\ref{fig:architecture} shows module relationships and synchronization primitives.

\subsection{Task queue}
The task queue wraps a \texttt{VecDeque<Task>} inside a \texttt{Mutex} and coordinates producers and consumers via a single \texttt{Condvar} \citep{rustmutex,rustcondvar}. \texttt{VecDeque} was chosen over a fixed-size ring buffer because the workload pushes a batch of tasks up front then drains them; a ring buffer would require a fixed capacity at construction with no practical benefit for this access pattern.

The critical section is minimal: a consumer holds the lock only long enough to inspect and pop the front element. When the queue is empty, \texttt{Condvar::wait} atomically releases the lock and suspends the thread until a producer calls \texttt{notify\_one}, avoiding busy waiting. Each task is dequeued at most once, satisfying the single-assignment invariant.

\noindent\textbf{Critical code path (blocking pop):}
\begin{lstlisting}
pub fn pop_blocking_or_closed(&self) -> Option<Task> {
    let mut guard = self.inner.lock().expect("task queue mutex poisoned");
    loop {
        if let Some(task) = guard.queue.pop_front() {
            return Some(task);
        }
        if guard.closed {
            return None;
        }
        guard = self.available.wait(guard).expect("condvar wait failed");
    }
}
\end{lstlisting}

The queue also exposes \texttt{try\_pop} for non-blocking access and a \texttt{close} operation that wakes all blocked consumers so threads can exit cleanly. Figure~\ref{fig:tq-flow} illustrates both producer and consumer paths.

Tests validate that tasks are consumed exactly once across multiple threads, that blocking consumers wake on push, and that closure unblocks all waiting threads. These tests target the invariant required by the project: single assignment of tasks under safe concurrent access.

\subsection{Zone access control}
The zone controller enforces exclusive occupancy per zone using a \texttt{Mutex<Vec<Option<RobotId>>>} and a vector of per-zone \texttt{Condvar}s \citep{rustmutex,rustcondvar}. A single shared condvar would cause unnecessary wakeups: when zone 1 is released, every waiting thread would wake and most would immediately re-block on their target zone. Per-zone condvars confine each \texttt{notify\_one} to the waiters for that specific zone, which reduces spurious contention when multiple zones are independently occupied.

Zones are 1-indexed and validated against the configured zone count, so each zone maps directly to one ownership slot and one condvar. This keeps the representation simple and removes hash-map lookup overhead from the hot path. When a robot finishes its work in a zone, it calls \texttt{release}, which clears the ownership slot and calls \texttt{notify\_one} on that zone's condvar, waking exactly one contender.

\noindent\textbf{Critical code path (acquire):}
\begin{lstlisting}
pub fn acquire(&self, zone: ZoneId, robot: RobotId) {
    let idx = self.zone_index(zone);
    let mut guard = self.occupied.lock().expect("zone mutex poisoned");
    loop {
        if guard[idx].is_none() {
            guard[idx] = Some(robot);
            return;
        }
        guard = self.condvars[idx].wait(guard).expect("condvar wait failed");
    }
}
\end{lstlisting}

Only one lock is held at a time, preventing deadlocks by construction. Unit tests spawn multiple contenders for a single zone and verify that the observed occupancy never exceeds one. Additional tests check that the occupied-zone snapshot matches current ownership and that out-of-range zone identifiers are rejected. Figure~\ref{fig:zone-flow} shows the acquire and release paths.

\subsection{Health monitor}
The health monitor uses one \texttt{Mutex<HealthState>} containing both a \texttt{HashMap<RobotId, Instant>} for heartbeat timestamps and a \texttt{HashSet<RobotId>} for the offline set \citep{rustmutex}. A two-lock design --- one lock per collection --- would introduce a window where a heartbeat refresh under the first lock is invisible to an offline check acquiring the second lock in a different order. A single mutex eliminates that race: \texttt{detect\_offline} reads heartbeats and updates the offline set in one critical section, so the snapshot is always coherent. The implementation relies on monotonic timestamps for timeout comparison \citep{rustinstant}.

\noindent\textbf{Critical code path (offline detection):}
\begin{lstlisting}
pub fn detect_offline(&self, timeout: Duration) -> HashSet<RobotId> {
    let now = Instant::now();
    let mut guard = self.inner.lock().expect("health monitor mutex poisoned");
    let overdue: Vec<_> = guard
        .heartbeats
        .iter()
        .filter_map(|(&robot, &last)| {
            if now.duration_since(last) > timeout { Some(robot) } else { None }
        })
        .collect();
    for robot in overdue {
        guard.offline.insert(robot);
    }
    guard.offline.clone()
}
\end{lstlisting}

\subsection{Synchronization strategy and invariants}
Each component uses a small, fixed set of synchronization primitives with a consistent lock order, eliminating deadlock cycles. The task queue and zone controller each center on a single mutex plus condition variables to coordinate waiting without busy loops. The zone controller uses direct-indexed per-zone condvars so that robots waiting on independent zones do not wake each other, reducing spurious contention. The health monitor uses a single \texttt{Mutex<HealthState>} so that heartbeats and offline detection update one coherent structure without cross-lock races. Lock scopes are kept tight to reduce contention. Atomics appear only for lightweight counters in the simulation harness and do not replace the fundamental locks protecting shared structures.

Each invariant maps to a specific code path:
\begin{itemize}[leftmargin=*]
  \item \textbf{Single task assignment}: \texttt{pop\_front} inside the mutex guard removes the task atomically; no second thread can see the same element.
  \item \textbf{Zone exclusivity}: the acquire loop checks and sets the ownership slot under the zone mutex; releasing under the same mutex before calling \texttt{notify\_one} prevents a new owner from entering before the old one has cleared the slot.
  \item \textbf{Offline detection}: the single-lock design in \texttt{detect\_offline} ensures the heartbeat snapshot and the offline update are atomic with respect to concurrent heartbeat calls.
  \item \textbf{Deadlock freedom}: no code path acquires more than one of the three module locks simultaneously, so there are no lock-ordering cycles.
\end{itemize}

\section{Benchmark}
\subsection{Methodology}
Benchmarks were measured on 2026-03-20 on macOS (Darwin 25.3.0, arm64) using \texttt{rustc 1.93.0}. Before collecting performance data, the required correctness gates were re-run: \texttt{cargo build --release} completed successfully and \texttt{cargo test} passed all 17 tests. All published numbers were then collected from the release binary (\texttt{./target/release/project\_blaze bench ...}) with the \texttt{validate} flag enabled, which keeps duplicate-task and zone-violation checks active during measurement.

To reduce coupling between rows, each published configuration was executed as five isolated benchmark runs rather than copied from one long stress sweep. The tables report mean \(\pm\) sample standard deviation for elapsed time, throughput, average zone wait, and CPU time. CPU user/system values come from a Unix \texttt{getrusage} delta and therefore support relative comparison on this machine rather than cross-platform claims. The \texttt{validate} flag activates a fixed seen-table keyed by task ID that detects any duplicate consumption across threads, and a \texttt{max\_occupancy} counter that records the highest simultaneous zone occupancy observed during the run; both checks add negligible overhead because the seen-table is pre-allocated to the total task count. The benchmark output also reports \texttt{zone\_violation}, \texttt{duplicate\_tasks}, and \texttt{offline\_robots}, so each timing result is paired with an explicit correctness signal. Offline-mode data is interpreted as ``at least one robot detected offline''; unlike the demo, the benchmark does not target a fixed robot identity.

\subsection{Interpretation}
The measurements support three conclusions. Correctness held throughout: duplicate-task detection never triggered, zone exclusivity never failed, and offline mode consistently reported one offline robot in the designated liveness scenario. Contention is the dominant performance bottleneck. With one zone, throughput stays near 162 tasks/s regardless of robot count because threads spend most of their time waiting to enter the single zone rather than doing work in parallel. With two zones, throughput roughly doubles at 2 and 4 robots, confirming that the synchronization design does not serialize independent zone accesses. Task-load growth scales almost linearly in elapsed time while throughput stays near 320 tasks/s for the 4-robot/2-zone configuration, which indicates that coordination costs are largely steady-state rather than startup-driven. These three patterns --- serialization under resource conflict, genuine parallelism when independent resources exist, and near-linear scaling in workload size --- are exactly what a minimal mutex-and-condvar design predicts.

The CPU-time columns support that interpretation. System time rises from 0.0016\,s in the baseline to 0.0038\,s in the 8-robot/4-zone case, consistent with more lock acquisitions and zone transitions rather than busy waiting. Offline mode is different: its average zone wait sits in the same range as the normal 4-robot/2-zone run, but total elapsed time expands because the harness deliberately waits for timeout confirmation after the work has completed. The standard deviations in Table~\ref{tab:bench-samples} are uniformly small relative to the means --- the largest relative deviation is the offline scenario at \(\pm\)48.53\,ms on a 707\,ms mean (6.9\%), which reflects variability in the liveness timeout window rather than lock contention. A cross-check with the CLI stress sweep showed the same qualitative pattern as the repeated single-run measurements, confirming that the tables capture stable behavior rather than a one-off schedule artifact.

\section{Discussion}
Correctness held across every scenario in the evaluation: no duplicate task assignments, no zone violations, and reliable offline detection in every offline run. That result is the central finding of the project, and it follows directly from the synchronization design rather than from luck. Each invariant is enforced at a single, identifiable code path --- the atomic pop-front inside the queue mutex, the slot-check-and-set inside the zone mutex, and the coherent heartbeat-plus-offline update inside the health monitor's single lock. When the code path is correct and the lock scope is tight, the invariant holds regardless of thread interleaving.

The zone-wait numbers in Table~\ref{tab:scalability} connect directly to the per-zone condvar design. With 4 robots and 1 zone, average wait climbs to 9284\,$\mu$s; with 4 robots and 2 zones, it drops to 3886\,$\mu$s. The improvement comes not just from having a second resource available, but from the notification structure: \texttt{notify\_one} on zone 1's condvar wakes exactly one thread waiting for zone 1, while threads waiting for zone 2 remain asleep. A single shared condvar would wake all waiters on every release, which adds overhead proportional to contention. In the current workload sizes that overhead is small, but the design principle --- targeted notification rather than broadcast --- is more correct at every scale. To illustrate, consider the 8-robot, 2-zone scenario in Table~\ref{tab:bench-samples}. When zone~1 is released, a shared \texttt{notify\_all} would wake all six blocked threads simultaneously. Each re-acquires the mutex, inspects its target zone, and either proceeds (one thread) or returns immediately to sleep (five threads). The five spurious wakeups add lock-acquisition and inspection overhead proportional to the number of waiters, scaling with contention rather than with useful work. Per-zone condvars eliminate this entirely: a release on zone~1 wakes at most one thread waiting for zone~1, while threads blocked on zone~2 are never disturbed.

The heartbeat timeout of 200\,ms in the demo was chosen to produce a visible offline event within the demo's runtime without requiring a long sleep. At 200\,ms, the monitor thread running every 50\,ms needs at most four detection cycles to observe the timeout after robot 1's last heartbeat. A shorter timeout (say 50\,ms) would risk false positives if a robot's thread is preempted for longer than the timeout window. A longer timeout would require a longer demo. In a real deployment the right value depends on the worst-case scheduling jitter and the acceptable detection latency, but the current design makes the parameter explicit and configurable, so the trade-off is transparent rather than buried in the implementation.

Rust's ownership model provides three concrete guarantees in this project. \texttt{Arc} ensures that the shared services (\texttt{TaskQueue}, \texttt{ZoneAccess}, \texttt{HealthMonitor}) are not freed while any robot thread still holds a reference; the reference count drops to zero only after every thread has joined. \texttt{Mutex} ensures that exactly one thread can access the protected data at a time; any thread that tries to read or write without the lock does not compile. \texttt{Condvar} pairs with \texttt{Mutex} to provide atomic ``release lock and sleep'' semantics; there is no window between releasing the lock and beginning the wait where a notification could be missed. Together, these three types replace the entire class of data-race and lost-wakeup bugs that would require manual inspection in C or C++.

The practical implication is that the correctness audit for MCRoCS does not require the manual lock-coverage analysis that an equivalent C implementation would demand. In C, a programmer verifies by inspection that every access to each shared variable is preceded by the correct lock acquisition --- a protocol the language neither expresses nor enforces. Rust inverts this relationship: the lock acquisition is what grants access to the data. A thread that reads or writes protected state without holding the lock produces a compile error, not a runtime data race. This shifts the primary verification burden from runtime observation to compile time, leaving only logical races --- such as whether the loop condition in \texttt{acquire} correctly handles spurious wakeups --- as the remaining correctness questions. Those questions are narrow enough to be answered by the unit tests rather than by exhaustive thread-interleaving analysis.

The benchmark has a notable limitation: each simulated task takes exactly 20\,ms of wall-clock sleep. Throughput numbers are therefore largely a function of that constant rather than of synchronization overhead. With 4 robots, 25 tasks, and 2 zones, the expected throughput ceiling is roughly $4 \times (1000/20) / (\mathrm{zone\ factor}) \approx$ 200--400 tasks/s depending on zone utilization, which matches the observed 320 tasks/s. If the work duration were reduced to 1\,ms, synchronization latency would become the bottleneck and the numbers would look very different. The current benchmark measures contention patterns correctly, but its absolute throughput figures should be interpreted in the context of the fixed 20\,ms work simulation.

Alternative synchronization primitives were considered. An \texttt{RwLock} would add mode-switching overhead for no benefit here because every queue operation, zone transition, and heartbeat update is a write; a queue pop is never safe to run concurrently with a push, so reader-writer splitting provides no throughput gain. Message-passing channels could replace the task queue --- each task becomes a channel message, and the receiving worker owns it exclusively from the moment of receive, eliminating the explicit pop-under-mutex pattern. However, channels would not remove the need for shared zone ownership or timeout bookkeeping, and introducing two coordination styles (channels for tasks, mutexes for zones and heartbeats) would fragment the mental model without a correctness benefit. The uniform \texttt{Mutex}+\texttt{Condvar} approach keeps the coordination story compact, easy to verify, and consistent across all three modules \citep{rustmutex,rustcondvar,herlihy2020}. The consistency also has a practical debugging benefit: when a correctness failure occurs in testing, the search space is bounded. Every invariant violation must originate in the module that owns that invariant's lock, because no other module can touch that state directly. This property --- fault isolation by lock ownership --- emerges from the architecture itself and would be lost if primitives were mixed or shared state crossed module boundaries.

\section{Conclusion and Future Work}
MCRoCS achieves its goal of implementing a lightweight OS core that safely coordinates multiple robot threads. The task queue, zone access controller, and health monitor each enforce a clear invariant --- single task assignment, exclusive zone occupancy, and heartbeat timeout detection --- and each invariant has a direct unit test. The demo shows all three behaviors in a deterministic, reproducible sequence, and the benchmark quantifies throughput and waiting behavior under contention with correctness checks active throughout. The 20-reference bibliography connects each major design choice to established OS theory and practitioner guidance, placing the implementation in the context of decades of concurrency research.

The minimal design was a deliberate choice. Mutexes and condition variables make the concurrency model explicit and easy to reason about at each call site. Short critical sections reduce contention without requiring lock-free data structures, which are harder to verify and test. The three-module architecture isolates each synchronization concern, which simplifies both correctness arguments and debugging: a failure in zone exclusivity cannot originate in health monitor code, and a missed heartbeat cannot corrupt the task queue. These properties follow the project's decision rule of correctness over performance, and they map cleanly onto the OS learning outcomes for concurrency control, synchronization, and coordination.

Future work could build on this foundation in several directions. First, longer and more varied performance runs would reduce measurement noise and let the benchmark isolate synchronization overhead from the fixed 20\,ms work simulation. Runs with sub-millisecond work durations would stress the lock paths directly and expose any contention that the current numbers obscure. Second, heterogeneous task durations and dynamic zone selection would produce a more realistic workload and stress the fairness properties of the per-zone condvar design; a task that holds a zone for ten times longer than average would dominate the wait-time distribution in a way the current benchmark does not capture. Third, the health monitor could adopt an adaptive timeout window --- widening when task throughput is low and narrowing when it is high --- to reduce false positives under temporary load spikes. An adaptive scheme could estimate an appropriate threshold from the rolling variance of recent inter-heartbeat intervals, setting the timeout at a fixed multiple of the measured jitter and adjusting automatically as the workload changes rather than requiring manual reconfiguration. Fourth, benchmark and stress modes could adopt deterministic liveness attribution similar to the demo, selecting a fixed target robot to stop heartbeats rather than relying on ``any offline'' outcomes; that would make the liveness timing results easier to interpret and compare across runs. It would also enable the benchmark to emit a precise detection-latency figure --- the interval between a robot's last heartbeat and its first appearance in the offline set --- giving a direct, quantitative measure of monitor responsiveness that the current output does not capture.

In its current form, the system is already suitable for the required demo and report deliverables: it has a clear architecture, observable concurrency behavior, and a test suite that directly verifies the three mandatory components. These qualities make it a sound baseline for any future extensions. The project demonstrates that a small set of well-chosen primitives, applied consistently, is sufficient to build a correct and demonstrable OS coordination layer. More broadly, it shows that the core OS concepts --- mutual exclusion, condition synchronization, and liveness detection --- are not abstract formalisms but engineering decisions with measurable consequences: a missing lock produces duplicate task consumption; a shared condvar inflates zone-wait latency under contention; a misjudged timeout either delays failure detection or raises false alarms. Keeping the system small enough to trace each consequence to its cause is the most instructive outcome the project can demonstrate.
